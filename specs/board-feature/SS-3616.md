# SS-3616: Board Feature — CRUD & List View

## Ticket Summary

**Type:** Story
**Status:** To Do
**Project:** (Dev) DevOps (SS)
**URL:** https://spotlio-team.atlassian.net/browse/SS-3616

Implement the board feature following the feature-sliced architecture. This ticket wires together the data layer delivered in SS-3615 and exposes it through React components and Next.js App Router pages. It covers Zod schemas for board mutation inputs, service functions over the existing Zustand store, client-side hooks, four React components, and two routed pages (`/boards` and `/boards/[boardId]/settings`). Full unit and component test coverage is required.

## Implementation Approach

### Phase 1 — Feature types

Create `src/features/board/types.ts`:

- `createBoardSchema`: `z.object({ title: z.string().min(1) })`
- `updateBoardSchema`: `z.object({ title: z.string().min(1) })`
- Export `CreateBoardInput` and `UpdateBoardInput` via `z.infer`

Note: Entity schemas (`BoardSchema`, `ColumnSchema`, etc.) already live in `src/features/board/types/index.ts` from SS-3615. The new `types.ts` holds only mutation-input schemas; it should re-export or import the entity types from `types/index.ts` as needed.

### Phase 2 — Service layer

Create `src/features/board/board.service.ts`. This module provides plain functions that wrap `useBoardStore` getState/dispatch calls, keeping components decoupled from the store internals:

- `getBoards(): Board[]`
- `getBoard(id: string): Board | undefined`
- `createBoard(input: CreateBoardInput): Board`
- `updateBoard(id: string, input: UpdateBoardInput): void`
- `deleteBoard(id: string): void`

All functions validate inputs against the Zod schemas and throw on failure.

### Phase 3 — Client hooks

Create `src/features/board/hooks/use-boards.ts`:
- Calls `useBoardStore` for reactive board list
- Triggers `loadBoards()` on mount (if not already loaded)
- Returns `{ boards, addBoard, deleteBoard }`

Create `src/features/board/hooks/use-board.ts`:
- Accepts `boardId: string`
- Selects the matching board from the store
- Returns `{ board, updateBoard, deleteBoard, isLoading }`

Both hooks must be `"use client"` directives at the top.

### Phase 4 — Components

All components live under `src/features/board/components/` and carry `"use client"`:

- `board-list.tsx` — renders `<BoardCard>` for each item; shows empty state when list is empty
- `board-card.tsx` — card with title, "Open" link to `/boards/[id]`, delete button
- `create-board-dialog.tsx` — uses a controlled input for title; calls `addBoard` on submit; closes on success or ESC
- `board-header.tsx` — accepts `boardId`, renders title with inline-edit toggle and a link back to `/boards`

### Phase 5 — Pages

`src/app/boards/page.tsx` (Server Component wrapper, client content via `use-boards`):
- Renders `<BoardHeader>` (without boardId context, just a title "My Boards")
- Renders `<CreateBoardDialog>` trigger button
- Renders `<BoardList>`

`src/app/boards/[boardId]/settings/page.tsx`:
- Receives `params.boardId`
- Renders `<BoardHeader boardId={boardId} />`
- Provides a form to rename the board and a danger-zone delete action

### Phase 6 — Barrel export

`src/features/board/index.ts` re-exports:
- All types from `./types` and `./types/index`
- All hooks from `./hooks/use-boards` and `./hooks/use-board`
- All components from `./components/*`
- `board.service` functions

### Phase 7 — Tests

Unit tests (`src/features/board/board.service.test.ts`):
- Mock `useBoardStore` and assert service delegates correctly
- Validate that invalid inputs are rejected by Zod

Component tests:
- `board-list.test.tsx` — renders empty state and populated list
- `board-card.test.tsx` — renders title, fires delete callback
- `create-board-dialog.test.tsx` — opens dialog, submits valid/invalid title
- `board-header.test.tsx` — renders title, fires rename, navigates back

## UI Components

### 1. `BoardList`

- **Purpose:** Renders the full list of boards fetched via `use-boards`. Shows an empty-state message when no boards exist.
- **Key props:** none (reads from hook internally) or `boards: Board[]` passed from the page
- **Key state:** none (stateless display component)
- **Layout/interaction:** Responsive grid of `<BoardCard>` items. Empty state centered with a call-to-action linking to `<CreateBoardDialog>`.

### 2. `BoardCard`

- **Purpose:** Displays a single board summary and provides navigation and delete actions.
- **Key props:** `board: Board`, `onDelete: (id: string) => void`
- **Key state:** none
- **Layout/interaction:** Card container with board title, an "Open" link (`/boards/[id]`), and a delete icon button. Delete triggers a confirmation or immediate call to `onDelete`.

### 3. `CreateBoardDialog`

- **Purpose:** Modal dialog for creating a new board.
- **Key props:** `onCreated?: (board: Board) => void`
- **Key state:** `open: boolean`, `title: string`, `error: string | null`
- **Layout/interaction:** Trigger button ("New board") opens a dialog. Dialog contains a text input for the title, a submit button, and a cancel button. On submit, validates with `createBoardSchema`; on success closes and optionally calls `onCreated`. ESC key closes the dialog. Uses a native `<dialog>` element or a headless approach — no external UI library assumed.

### 4. `BoardHeader`

- **Purpose:** Page header for the board settings page (and optionally the list page).
- **Key props:** `boardId?: string`, `title?: string`
- **Key state:** `isEditing: boolean`, `draftTitle: string`
- **Layout/interaction:** Displays the board title. When `boardId` is provided, shows an edit icon that toggles an inline text input to rename the board. Calls `updateBoard` on blur or Enter. Always shows a "Back to boards" link navigating to `/boards`.

### 5. `/boards` page (`BoardsPage`)

- **Purpose:** Full-page list view of all boards.
- **Key props:** none (Next.js page, no external props)
- **Key state:** Delegated to `use-boards`
- **Layout/interaction:** Page-level layout with a heading "My Boards", a "New board" button (opens `<CreateBoardDialog>`), and the `<BoardList>`. Uses `"use client"` because it interacts with the store.

### 6. `/boards/[boardId]/settings` page (`BoardSettingsPage`)

- **Purpose:** Settings page for a specific board.
- **Key props:** `params: { boardId: string }` (Next.js dynamic route)
- **Key state:** Delegated to `use-board`
- **Layout/interaction:** `<BoardHeader boardId={boardId} />` at top. Below: a rename form (controlled input pre-filled with current title) and a "Danger Zone" section with a delete board button. On delete, navigate back to `/boards`.

## Risks

1. **Zustand store not pre-loaded on navigation.** The `use-boards` and `use-board` hooks must call `loadBoards()` on mount; if omitted the store is empty on hard refresh.
   - Mitigation: Call `loadBoards()` in a `useEffect` inside each hook with an empty-dependency guard.

2. **`"use client"` boundary leakage.** Next.js 15 App Router requires explicit `"use client"` markers for components that use hooks. Missing markers will cause server-render errors.
   - Mitigation: All hook-using components and pages carry `"use client"` at the top.

3. **Zod v4 API changes.** The installed `zod@^4.3.6` is the v4 branch, which has minor breaking API differences from v3 (e.g., `.safeParse` return shape is unchanged, but some utility types differ). The existing `types/index.ts` already uses v4 syntax, so the new schemas must remain consistent.
   - Mitigation: Follow the patterns in `src/features/board/types/index.ts` exactly.

4. **Barrel export circular dependency.** If `index.ts` imports from `board.service.ts` and the service imports from `types/index.ts`, and `types/index.ts` is also re-exported from `index.ts`, a circular import can form.
   - Mitigation: Service and hooks must import from `./types/index` directly, not from the barrel.

5. **Test environment lacks `localStorage`.** Vitest runs in jsdom (configured in SS-3614). The service tests will need to reset `localStorage` between tests to avoid cross-test contamination.
   - Mitigation: Call `localStorage.clear()` in `beforeEach` or mock the storage module.

6. **No routing guard.** If a user navigates to `/boards/[boardId]/settings` with an invalid `boardId`, `use-board` returns `undefined`. The page must handle this gracefully with a 404 or redirect.
   - Mitigation: Add a `notFound()` call (Next.js) when `board` is `undefined` after store load.

## Open Questions

1. **Dialog implementation.** Should `CreateBoardDialog` use the HTML `<dialog>` element, a Radix UI primitive, or a custom implementation? No UI component library is listed in `package.json`.
   - Default assumption: native `<dialog>` with CSS, unless the ticket author specifies otherwise.

2. **Board settings scope.** The ticket mentions a settings page but does not specify what settings exist beyond renaming. Are there other settings (visibility, description, background color) intended for this ticket or deferred?
   - Default assumption: rename only, plus delete, for this ticket.

3. **Navigation from `BoardCard`.** The "Open" link goes to `/boards/[id]` — but there is no board detail/kanban view page defined in this ticket. Should the link be disabled, go to settings, or be omitted until a future ticket?
   - Default assumption: Link to `/boards/[id]/settings` for now; the kanban view will be added in a later ticket.

4. **`BoardHeader` on the list page.** The list page (`/boards`) does not have a `boardId`. Should `BoardHeader` be used on this page or a simpler heading element?
   - Default assumption: Use a plain `<h1>` heading on the list page; `BoardHeader` is only used on the settings page.

5. **Confirmation on delete.** Should `BoardCard` and the settings page show a confirmation prompt before deleting a board?
   - Default assumption: `window.confirm` or an `aria`-accessible inline confirmation, matching the project's minimal UI stance.
