Implement the **column feature** as a standalone feature slice under `src/features/column/`. The scope covers Zod-based mutation schemas, a service layer for CRUD operations and drag-to-reorder logic (using the existing gap-strategy position utilities), a React hook for column state, and three UI components: `Column`, `ColumnHeader`, and `AddColumn`. Columns belong to a board; deleting a column must cascade-delete its cards and associated card-labels. Renaming is done via inline edit. The feature follows the same file conventions established by the board feature in SS-3616.

### Phase 2 -- Service Layer

**Files:**
- `src/features/column/column.service.ts`
- `src/features/column/column.service.test.ts`

**Notes:**
- Follow the same pattern as `board.service.ts`: thin functions that validate input with Zod, then delegate to `useBoardStore.getState()`.
- `getColumns(boardId)` -- returns columns filtered by boardId, sorted by position.
- `getColumn(id)` -- returns single column or undefined.
- `createColumn(boardId, input)` -- parse with `createColumnSchema`, call `store.addColumn(boardId, title)`, return the new column.
- `updateColumn(id, input)` -- parse with `updateColumnSchema`, call `store.updateColumn(id, title)`.
- `deleteColumn(id)` -- call `store.deleteColumn(id)`. The store already handles removing the column from localStorage. Cascade deletion of cards must also be triggered (the store's `deleteColumn` currently only removes the column; cards in that column and their card-labels must be cleaned up -- see Risks).
- `reorderColumn(id, beforePos?, afterPos?)` -- call `store.moveColumn(id, beforePos, afterPos)`.
- Tests should use the Zustand store directly (reset between tests) and validate CRUD + position reindexing.

### Phase 3 -- Hook

## UI Components

### 1. Column (`column.tsx`)

- **Purpose**: Renders a single kanban column lane in the board view. Acts as a container for the column header and a scrollable area where cards will be rendered as children.
- **Key Props**:
  - `column: Column` -- the column entity (id, boardId, title, position, createdAt)
  - `onRename: (id: string, title: string) => void` -- callback for renaming
  - `onDelete: (id: string) => void` -- callback for deletion
  - `onDragStart: (e: DragEvent, columnId: string) => void` -- drag handle callback
  - `onDragOver: (e: DragEvent) => void` -- drag target callback
  - `onDrop: (e: DragEvent, targetColumnId: string) => void` -- drop callback
  - `children: ReactNode` -- card slots rendered inside the column body
- **Key State**:
  - `isDragOver: boolean` -- highlight state when another column is dragged over this one
- **Layout / Interaction Notes**:
  - Vertical column layout with fixed width (~280px), full board height minus header.
  - Background: `#171a1f` with glassmorphism box-shadow (`0 20px 40px rgba(129,236,255,0.08)`), 4px border-radius.
  - Drag-over visual: subtle border highlight (`#81ecff` at low opacity) when `isDragOver` is true.
  - Column body scrolls vertically when cards overflow.
  - Column itself is `draggable`; drag handle is in the header area.

### 2. ColumnHeader (`column-header.tsx`)

- **Purpose**: Displays the column title with inline-edit capability and a delete button. This is the primary interaction point for renaming and removing columns.
- **Key Props**:
  - `title: string` -- current column title
  - `columnId: string` -- column ID for callbacks
  - `onRename: (id: string, title: string) => void` -- commit rename
  - `onDelete: (id: string) => void` -- trigger delete with cascade confirmation
  - `onDragStart?: (e: DragEvent, columnId: string) => void` -- drag handle
- **Key State**:
  - `isEditing: boolean` -- toggles between display and edit mode
  - `editValue: string` -- controlled input value during edit
- **Layout / Interaction Notes**:
  - Flex row: drag handle (grip icon or cursor:grab) | title text | delete button.
  - Title text uses Space Grotesk, 16px, weight 600, color `#f6f6fc`. Clicking the title enters edit mode.
  - Edit mode: replaces title text with an `<input>` (Inter, 16px, transparent bg, `#81ecff` bottom border). Auto-focuses and selects all text on enter.
  - Enter or blur commits the rename (if non-empty; otherwise reverts). Escape cancels and reverts.
  - Delete button: trash SVG icon (same as `board-card.tsx`), `#aaabb0` default, `#ff716c` on hover. Shows `window.confirm` with cascade warning: "Delete column [title]? All cards in this column will also be deleted."
  - Drag handle: a grip/dots icon or the entire header area. Sets `cursor: grab` (and `cursor: grabbing` while dragging).

### 3. AddColumn (`add-column.tsx`)

- **Purpose**: Provides a compact UI element at the end of the column row for adding a new column. Starts as a `+` button and expands into a text input on click.
- **Key Props**:
  - `onAdd: (title: string) => void` -- callback when a new column title is submitted
- **Key State**:
  - `isExpanded: boolean` -- toggles between button mode and input mode
  - `title: string` -- controlled input value
  - `error: string | null` -- validation error (empty title)
- **Layout / Interaction Notes**:
  - Collapsed state: a rectangular button (~280px wide, same width as column lanes), dashed border (`#aaabb0` at low opacity), `+` icon and "Add column" text, Inter 14px, `#aaabb0` color. Hover: border color shifts to `#81ecff`.
  - Expanded state: replaces button with a compact form. Input field (Inter, 14px, transparent bg, `#81ecff` bottom border) + small "Add" submit button (primary style matching `CreateBoardDialog`) + "Cancel" text button.
  - Enter submits; Escape collapses back to button mode.
  - Auto-focus input on expansion.
  - After successful creation, collapse back to button and clear input.

## Risks

1. **Cascade deletion is incomplete in store**: The existing `deleteColumn` action in `board-store.ts` (lines 185-195) only removes the column from state and persists the updated columns list. It does NOT cascade-delete cards belonging to that column or their card-labels from localStorage. The `deleteBoard` action does handle cascading (lines 109-139), so the pattern exists.
   - **Mitigation**: Before wiring the service `deleteColumn`, enhance the store's `deleteColumn` action to cascade-delete cards (via `removeKey(STORAGE_KEYS.cards(column.id))`) and associated card-labels. Alternatively, handle cascade in the service layer. The store approach is preferred for consistency with `deleteBoard`.

2. **No DnD library installed**: Drag-to-reorder uses HTML5 Drag and Drop API, which has known quirks (no touch support, limited drag preview customization, inconsistent behavior across browsers on mobile).
   - **Mitigation**: Implement basic desktop DnD with HTML5 API. Note in code comments that a future ticket may replace this with `@dnd-kit` or similar. Touch/mobile reordering is out of scope for this ticket.

3. **Inline edit accessibility**: Click-to-edit patterns can be problematic for screen readers and keyboard navigation if not implemented carefully.
   - **Mitigation**: Use `role="textbox"` and `aria-label` on the display element, ensure the edit input is properly labeled, and support full keyboard interaction (Enter, Escape, Tab).

4. **Position collision during rapid reordering**: If a user drags columns very rapidly, the gap between positions can shrink below the MIN_GAP threshold before `needsReindex` triggers.
   - **Mitigation**: The existing `needsReindex` / `reindexPositions` logic in the store's `moveColumn` handles this correctly. Verify with tests that reindexing fires when gaps are exhausted.

## Open Questions

1. **Should the board detail page (`/boards/[boardId]`) be created in this ticket or a future one?**
   - Default assumption: No. This ticket builds the column feature slice only. A separate ticket will wire the board detail page that renders columns. Components are built for composition.

2. **Should drag-to-reorder support touch devices?**
   - Default assumption: No. HTML5 DnD is desktop-only. Touch support deferred.

3. **Should there be a maximum number of columns per board?**
   - Default assumption: No limit enforced in this ticket. If needed, a validation rule can be added to `createColumnSchema` or the service layer later.

4. **What should happen if the user tries to delete the last remaining column?**
   - Default assumption: Allow deletion. The board can have zero columns. The UI should handle the empty state gracefully.

5. **Should column title length be constrained?**
   - Default assumption: Enforce `z.string().min(1).max(100)` in the schema to prevent excessively long titles from breaking layout.
