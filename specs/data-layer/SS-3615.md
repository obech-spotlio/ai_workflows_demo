# SS-3615: Data Layer with localStorage Persistence

## Ticket Summary

Implement a type-safe data layer for a kanban board application using Zod schemas, TypeScript strict typing, localStorage persistence, and Zustand for reactive state management. The data layer will support Board, Column, Card, and Label entities with position management using a gap strategy (increments of 1000).

**Key Deliverables:**
- Zod schemas and inferred TypeScript types for all entities
- localStorage service with typed CRUD operations
- Position management utility with gap strategy and re-indexing
- Zustand store for reactive state management
- Unit tests covering storage service and position logic

## Acceptance Criteria

- [ ] Zod schemas defined for all entities: Board, Column, Card, Label, CardLabel
- [ ] TypeScript types inferred from Zod schemas
- [ ] localStorage service with typed read/write for all entities
- [ ] Position management with gap strategy (increments of 1000, re-index when gap < 1)
- [ ] Zustand store for reactive state management
- [ ] Unit tests for localStorage service and position management logic

Generate TypeScript types using Zod's type inference (`z.infer<typeof Schema>`).

### Phase 3: localStorage Service

Create `src/shared/lib/storage/storage.ts` with:

**Core CRUD Operations:**
- `read<T>(key: string): T | null` - Read typed data from localStorage
- `write<T>(key: string, data: T): void` - Write typed data to localStorage
- `remove(key: string): void` - Remove data from localStorage
- `clear(): void` - Clear all application data

**Entity-Specific Methods:**
- `readBoards(): Board[]`
- `writeBoard(board: Board): void`
- `readColumns(boardId: string): Column[]`
- `writeColumn(column: Column): void`
- `readCards(columnId: string): Card[]`
- `writeCard(card: Card): void`
- `readLabels(boardId: string): Label[]`
- `writeLabel(label: Label): void`
- `readCardLabels(cardId: string): CardLabel[]`
- `writeCardLabel(cardLabel: CardLabel): void`

**Validation:**
- All reads/writes validate against Zod schemas
- Throw descriptive errors on validation failure
- Support hydration from corrupted localStorage (fallback to empty state)

### Phase 4: Position Management Utility

Create `src/shared/lib/storage/position.ts` with:

**Gap Strategy Logic:**
- Positions are increments of 1000 (1000, 2000, 3000, etc.)
- When inserting between existing items, use midpoint (e.g., between 1000 and 2000 → 1500)
- When gap between two positions < 1, trigger full re-indexing

**Functions:**
- `calculatePosition(before?: number, after?: number): number`
  - Returns next available position based on surrounding items
  - Handles edge cases: start of list, end of list, between items
- `reindexPositions(items: T[]): T[]`
  - Re-indexes all items to multiples of 1000 (1000, 2000, 3000, ...)
  - Maintains original order
  - Used when gap between positions < 1

**Unit Tests:**
- Test position calculation: start, end, between, multiple levels
- Test re-indexing: maintains order, resets to 1000 increments
- Test edge cases: single item, duplicate positions

## Constraints and Assumptions

**Project Constraints:**
- TypeScript strict mode enabled
- Feature-sliced directory structure must be followed
- All code must be unit tested with vitest
- No external database; localStorage is the source of truth for this phase

**Data Model Assumptions:**
- Board IDs, Column IDs, Card IDs, and Label IDs are UUIDs (use crypto.randomUUID())
- Position is a number; gap strategy ensures sufficient "space" for drag-and-drop operations
- CardLabel is a many-to-many junction; a card can have multiple labels
- createdAt is immutable (never updated); updatedAt tracks modifications
- Color for labels is a hex string (e.g., "#FF5733")
- localStorage key namespacing: `board:<key>` (e.g., `board:boards`, `board:columns:${boardId}`)

**Browser Compatibility:**
- localStorage available in modern browsers; no special polyfills needed
- Assume users have sufficient quota (5-10MB typically available)

## Risks and Open Questions

### Risks

1. **localStorage Quota Exceeded**: If user creates many boards/cards, localStorage could fill up
   - Mitigation: Document quota expectations; consider implementing cleanup for archived boards later

2. **Data Corruption**: User could modify localStorage manually or browser cache could fail
   - Mitigation: Zod validation on all reads; fallback to empty state on corruption; consider version migration strategy

3. **Concurrent Tabs**: Multiple browser tabs with same origin could cause race conditions
   - Mitigation: Plan for storage events listener (phase 2); current scope assumes single tab

4. **Position Calculation Edge Case**: Re-indexing logic must preserve order and handle all items correctly
   - Mitigation: Comprehensive unit tests with edge cases (single item, duplicates, boundary conditions)

### Open Questions

1. **Migration Path**: How will schema versioning handle migrations when entities change in future phases?
   - Proposed: Add schema version to localStorage, implement migration functions later

2. **Offline Sync**: Should this phase support syncing with a backend when available, or is localStorage the permanent store?
   - Current: localStorage as source of truth; sync logic deferred to future phases

3. **Performance**: With many cards (1000+), should we implement any indexing or pagination in the store?
   - Current: Load all data into memory; optimize if performance issues arise

4. **Undo/Redo**: Should the store track history for undo operations?
   - Current: Not in scope; can be added as a feature later with middleware

5. **Real-time Collaboration**: Multi-user editing in same board is not supported in this phase
   - Current: Single-user only; client-side state management
